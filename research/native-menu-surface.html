Native Menu Surface Research

This research track exists to answer one question: how can a Codex plugin present Plan-style integrated input controls for the GH Project workflow?

Strongest Current Finding

MCP elicitation is the leading implementation path. A plain Skill cannot create the integrated menu. A plugin manifest also does not define arbitrary composer menus. A plugin-bundled MCP server can request structured user input during a tool call through MCP elicitation, and Codex source contains app-server protocol types plus a round-trip test for that request.
The protocol feasibility probe passes. The repository now contains a minimal plugin-bundled MCP server with one tool, preview_gh_project_menu, that asks for a single .gitignore choice using MCP form elicitation. A deterministic SDK client receives the elicitation request, accepts gitignoreTemplate=none, and the tool returns that selected value.
Desktop visual proof failed. Codex Desktop called gh-project-plugin.preview_gh_project_menu, but returned {"action":"decline"} with no native integrated input-control menu. This falsifies the working assumption that the current Desktop build renders arbitrary MCP form elicitation as a Plan-style menu.
Current blocker: host UI support in Codex Desktop. Plugin discovery is resolved, the MCP server is callable, and the elicitation request is valid under the local SDK round trip. The observed Desktop behavior is an internal decline instead of a rendered menu.
Applied fix for the GH Project workflow: enable Codex's native Default-mode input surface instead of forcing this through MCP elicitation. /Users/velocityworks/.codex/config.toml now has default_mode_request_user_input = true and auth_elicitation = true. Codex Desktop must be restarted before an already-running chat can use that new tool availability.
Native tool discovery is now proven. The MCP server is registered in ~/.codex/config.toml as gh-project-plugin, with cwd pinned to ~/.codex/plugins/cache/local/gh-project-plugin/0.1.0. codex mcp list shows it enabled, and fresh Codex sessions call preview_gh_project_menu through the native MCP tool path.

Concrete Answers

Can a Skill render the menu? No. A Skill is workflow memory and agent instructions. It can tell Codex to use a tool, but it cannot expose a new host UI surface.
Can plugin.json render the menu? No evidence found. The official source manifest supports plugin metadata, skills, MCP servers, apps, hooks, and interface assets/prompts. It does not define a general menu or form surface.
Why did request_user_input fail? The handler checks the current collaboration mode against its allowed modes. In Default mode it returns request_user_input is unavailable in Default mode.
What native path exists? MCP elicitation. The MCP spec defines server-requested structured input. Codex implements MCP elicitation client handling and app-server request/response protocol types.
What must be proved next? The Desktop visual probe is complete and failed for MCP elicitation. For the GH Project workflow, the implemented route is now Default-mode request_user_input after restarting Codex Desktop so the updated feature flags are loaded.

Evidence

Official / primary documentation

  • OpenAI Academy states that plugins connect Codex to tools and sources of information, while skills are playbooks for process: Plugins and skills.
  • MCP elicitation defines structured user input requested by servers through the client: MCP Elicitation specification.

OpenAI Codex source evidence

  • Plugin manifest parser: codex-rs/core-plugins/src/manifest.rs. Observed supported surfaces: skills, mcpServers, apps, hooks, and interface.
  • request_user_input handler: request_user_input.rs. The handler rejects calls when the current collaboration mode is not in the allowed mode list.
  • request_user_input schema: request_user_input_spec.rs. The schema matches the Plan-style choice UI: short questions, 2 to 3 choices, recommended option first, and automatic free-form Other.
  • Plan mode template: plan.md. It explicitly instructs the model to prefer request_user_input.
  • Collaboration mode presets: collaboration_mode_presets.rs. The app currently defines built-in Plan and Default presets.
  • MCP elicitation app-server test: mcp_server_elicitation.rs. The test expects a ServerRequest::McpServerElicitationRequest and resolves it with McpServerElicitationRequestResponse.
  • MCP elicitation client service: elicitation_client_service.rs. Codex handles CreateElicitationRequest from MCP servers and returns accept/decline/cancel responses.
  • TUI elicitation renderer: mcp_server_elicitation.rs. It renders select and text fields as a bottom-pane form surface.

Upstream issue/discussion evidence

  • openai/codex discussion #11717 proposes custom collaboration modes because request_user_input is currently restricted to Plan mode. This is not official documentation, but it matches the observed runtime behavior and explains the limitation.

Acceptance criteria

  • The user sees a host-native input-control UI, not Markdown.
  • The UI includes at least two choices and a recommended/default value.
  • The MCP tool receives the selected value and returns structured JSON.
  • No repository is created during the probe.

Invalidated Paths

Do not attempt to solve this by adding more prose to SKILL.md. The Skill can document the workflow, but it cannot by itself cause Codex Desktop to render native menus in Default mode.

Do not rely on MCP elicitation alone for the GH Project native menu in the current Desktop build. The server-side request is valid, but Desktop declines it instead of presenting a user-facing control.

GH Project Plugin Specification

The plugin creates new GitHub repositories with low-friction defaults, native integrated menu choices, and explicit confirmation before any external or filesystem mutation.

Purpose

The plugin turns a request such as Create a new GitHub project for Apple Mail Handler into a complete repository creation workflow. It infers the likely setup, presents a native choice flow for review or edits, and creates the GitHub repository plus local clone only after explicit approval.

In this plugin, GitHub project means a GitHub repository by default. It does not mean a GitHub Projects board unless the user explicitly asks for one.

Default Contract

Owner Use the active authenticated gh account.
Location ~/IdeaProjects, expanded before collision checks.
Visibility Public.
License MIT.
Initial files README and MIT license. No scaffold unless requested.
Repo features Leave GitHub defaults unchanged. Do not toggle issues, wiki, projects, or discussions unless requested.
Description Generate plain searchable text capped at 350 characters.
Topics Generate 6 to 10 GitHub repository topics optimized for user search queries.
.gitignore Ask every time. Recommend a template only when the implementation stack is clear.

Menu Behavior

The menu must be host-native. The plugin must use the integrated input-control surface shown in Codex planning workflows. A numbered Markdown list is not an acceptable substitute.

Required interaction pattern

  1. Infer a complete creation plan before asking the user to fill fields.
  2. Present every configurable setting in the integrated input-control UI.
  3. Preselect inferred defaults instead of hiding them.
  4. Ask one setting at a time.
  5. Show the final repo creation plan before mutation.
  6. Create the repository only after the user selects Create GitHub project.

Required setting menus

  • Parent location.
  • Repo, folder, and friendly name.
  • Description.
  • Topics.
  • Visibility.
  • License.
  • Initial contents.
  • .gitignore.

Required final choices

  • Create GitHub project
  • Change settings
  • Do something else

Native menu requirement

The plugin must expose its choices through a plugin or tool surface that Codex Desktop can render as integrated input controls. If that surface is unavailable, the workflow must stop and state the blocker. It must not degrade to prose menus, Markdown numbered lists, or hidden defaults.

Low friction means defaults are preselected, not hidden. The user should be able to press the recommended choice repeatedly and move quickly through the wizard while still seeing every setting before creation.

Execution Contract

The plugin uses one deterministic creation path through the GitHub CLI. It performs read-only checks before mutation, then runs the creation commands only after explicit confirmation. 

gh repo create REPO --public --clone --add-readme --license mit --description "DESCRIPTION"
gh repo edit OWNER/REPO --add-topic topic-a,topic-b,topic-c

Include --gitignore TEMPLATE only when the user selected a GitHub gitignore template. Replace --public or --license mit only when the user changed those values through the native menu.

Stop Conditions

  • gh is missing.
  • gh is unauthenticated.
  • The active GitHub owner cannot be resolved.
  • The target local folder already exists.
  • The target GitHub repository already exists.
  • The user asks for a GitHub Projects board instead of a repository.
  • The integrated input-control UI is unavailable when a menu decision is required.
  • The user has not explicitly selected Create GitHub project.

Native Menu Surface

The working implementation path is the Codex native request_user_input surface in Default mode. The required local feature flag is enabled in /Users/velocityworks/.codex/config.toml with default_mode_request_user_input = true. Codex Desktop must be restarted after this config change before the menu surface is available to normal chats. 

Skill text = workflow contract
request_user_input = native integrated menu
